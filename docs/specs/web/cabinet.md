> Статус: ✅ Реализовано — 26 февраля 2026
> Автор: @веб
> Дата: 26 февраля 2026

---

## Страница

**Название:** Личный кабинет (Client Cabinet)

**URL:** `/cabinet` (корень), `/cabinet/billing`, `/cabinet/connections`, `/cabinet/settings`

**Тип:**
- [x] Страница с данными + управление
- [x] Приватная (нужен JWT)

---

## Авторизация

**Нужна:** да

**Механизм:** JWT в httpOnly cookie (тот же что при логине)

**Если нет токена:** редирект на `/login.html?redirect=/cabinet`

---

## Аудитория

Действующий клиент Аркентия — владелец или операционный директор сети доставки. Уже подключён, оплатил подписку (или на триале). Заходит чтобы:
- Проверить статус подключений
- Управлять точками и модулями
- Посмотреть платежи и счета
- Изменить настройки

**Устройство:** десктоп первичен, мобайл — адаптивен но не приоритет.

## Пустые состояния

| Экран | Пустое состояние |
|-------|-----------------|
| Обзор — нет событий | «Пока нет событий. Они появятся после первого отчёта» |
| Подписка — триал | Жёлтая плашка: «Триал до DD.MM.YYYY. Оплатите подписку чтобы продолжить» + кнопка «Оплатить» |
| Подключения — не настроен iiko | Серая карточка: «iiko не настроен» + кнопка «Настроить» |
| Платежи — пусто | «Платежей пока нет» (триал) |
| Загрузка | Skeleton-loader на карточках и таблицах |
| Ошибка API | Toast: «Не удалось загрузить данные. Попробуйте позже» |

## Стек и ограничения

**Фронтенд:** HTML + Tailwind + Vanilla JS (MVP, как остальные страницы)

Аргументация: кабинет — набор карточек и таблиц с CRUD-операциями. Для MVP не нужен React. Если вырастет до сложного SPA с real-time обновлениями — можно мигрировать позже.

**Адаптивность:** desktop-first, мобайл адаптивен (sidebar → hamburger или bottom nav)

**Браузеры:** Chrome, Safari (последние 2 версии)

**Навигация:** multi-page (каждый раздел — отдельный HTML), sidebar общий (include через JS)

## Заметки для @интегратора

- Все `/api/cabinet/*` endpoints — только с валидным JWT (401 без токена)
- tenant_id берётся из JWT payload, не из URL (безопасность)
- Prorating при изменении подписки: считать пропорционально оставшимся дням до следующего платежа
- Health-check iiko: отдельный cron job, результат в БД, API читает из кэша
- PDF чеков/актов: генерировать лениво (по запросу) или хранить готовые
- chain.json для добавления точек: тот же `/data/chain.json` что в визарде

**Мультитенантность TG-чатов:**
- Таблица `telegram_chats`: `id`, `tenant_id`, `chat_id`, `name`, `cities` (JSON), `modules` (JSON), `is_active`
- `GET /api/cabinet/chats` — возвращает только чаты где `tenant_id = jwt.tenant_id`
- `POST /api/cabinet/chats/verify` — проверяет через Telegram Bot API что бот добавлен в указанный chat_id (sendChatAction или getChat)
- Один chat_id может принадлежать только одному tenant (уникальный индекс по chat_id)
- При удалении чата — `is_active = 0`, не физическое удаление (для аудита)
