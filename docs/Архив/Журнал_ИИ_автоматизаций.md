## 25.02.2026 — Аудит: живые unclosed, on-demand генерация, формат отмен

### Контекст
Незакрытые заказы (`unclosed_in_transit`) отображались из статичного снимка `audit_events`, а не из реального состояния `orders_raw`. Причины отмен и тип оплаты не сохранялись при Phase 2 cleanup. Аудит за прошлые даты работал только если job уже запускался.

### Что сделано

**1. unclosed_in_transit — живая генерация (`audit.py`):**
- Вынесен в отдельную функцию `_detect_unclosed_in_transit(date_str)`
- Больше НЕ сохраняется в `audit_events` (транзиентное состояние)
- Генерируется живо при каждом вызове: ежедневный job и команда `/аудит`
- Поля date/city заполняются прямо в функции

**2. On-demand генерация аудита (`audit.py`):**
- `handle_audit_command`: если `get_audit_events()` пуст для даты — вызывает `_generate_audit_for_date()` на лету
- Результат сохраняется в `audit_events` как кэш для повторных запросов
- `/аудит Черногорск 14.02` — работает даже если daily job не запускался за эту дату
- Новый хелпер `_generate_audit_for_date()` — инкапсулирует всю детекцию + дедупликацию

**3. Формат отмен — расширенная информация (`audit.py`):**
- Новый формат: `#51721 — 2 347₽ отменён | без оплаты | не готовился | Отказ гостя`
- `payment_type`: тип оплаты или "без оплаты"
- `cooked_time`: маркер "готовился" / "не готовился" (по наличию cooked_time)
- `cancel_reason`: явное "без причины" если пусто
- `comment`: строка-подсказка `└ текст комментария` если есть в заказе

**4. cancel_sync: причина и оплата при Phase 2 (`cancel_sync.py`):**
- `_fetch_cancelled_from_server`: добавлен `PayTypes` в OLAP-запрос
- Phase 1: второй UPDATE для заказов уже `Отменена` с пустой `cancel_reason` — дозаполняет причину
- Phase 2: `extra_cancelled` теперь dict с данными (cancel_cause, payment_type), не просто set
- UPDATE включает `cancel_reason` и `payment_type` через `COALESCE(NULLIF(?, ''), existing)`

**5. Бэкфил за февраль 2026:**
- Одноразовый скрипт через OLAP: 902 отменённых заказа найдено, 168 получили cancel_reason
- Все 100% февральских отмен теперь с причиной и типом оплаты

### Файлы
- `app/jobs/audit.py` — живой unclosed, `_generate_audit_for_date`, on-demand, формат отмен
- `app/jobs/cancel_sync.py` — PayTypes в OLAP, Phase 1 backfill, Phase 2 с причинами

## 25.02.2026 — Аудит: дедупликация сторно/отмена, зависшие заказы

### Контекст
Заказ #85998 отображался одновременно в "Сторно + скидка" и "Отменённые" в `/аудит`. Заказы типа #119373 висели в статусе "В пути к клиенту" спустя дни после реальной отмены.

### Что сделано

**1. Дедупликация: отмена приоритетнее сторно (`audit.py`):**
- Если заказ присутствует и как `cancellation` (из `orders_raw`), и как `storno_discount` (из OLAP) — остаётся только в "Отменённые"
- Логика: собираем `cancelled_nums`, фильтруем `storno_findings` по ним
- Ранее приоритет был у сторно → отменённые заказы дублировались

**2. Тег филиала [Б1] перед номером заказа (`audit.py`):**
- `_tag_description()` теперь ставит тег ДО номера: `[Б1] #119861 ...`
- Для сторно: убирает встроенное имя филиала, подставляет короткий тег
- Единообразный формат во всех секциях аудита

**3. cancel_sync: очистка зависших заказов (Phase 2):**
- Новая фаза после обычной синхронизации отмен
- Находит заказы в нефинальных статусах (`Новая`, `В пути к клиенту` и пр.) старше 1 дня
- Запрашивает OLAP на предмет отмен в диапазоне дат зависших заказов
- Обновляет: найден в отменённых → `Отменена`, иначе → `Закрыта`
- Результат: 161 → 0 зависших заказов при первом запуске, 65 → 2 (вчерашние) после уменьшения cutoff с 2 до 1 дня

## 25.02.2026 — Бэкфил daily_stats + исправление avg_wait_min

### Контекст
`/отчёт барнаул 14.02` показывал пустые значения: скидки "—", себестоимость "—", опоздания "0 из 500 доставок" (неправильно), времена отсутствуют. Причины: daily_stats для дат до 21.02 содержал NULL в новых полях; `avg_wait_min` всегда NULL из-за бага в формуле; `total_delivered` = `orders_count` вместо реального числа доставок.

### Что сделано

**1. Исправлен `avg_wait_min` (баг):**
- Формула считала `send_time - ready_time`, но `ready_time` никогда не заполняется OLAP-обогащением (в OLAP нет поля ReadyTime)
- Заменено на `send_time - cooked_time` (ожидание от "приготовлен" до "отправлен курьеру")
- Исправлено в обеих функциях: `aggregate_orders_for_daily_stats` и `aggregate_orders_today`

**2. `/отчёт [дата]` — живая агрегация для одиночных дат:**
- Раньше брал все данные из `daily_stats` (если NULL — показывал "—")
- Теперь для одиночных дат вызывает `aggregate_orders_for_daily_stats()` напрямую из `orders_raw`
- `stats` (revenue, checks, cogs) по-прежнему из `daily_stats`, но `agg` (times, delays, staff, discounts) — всегда свежие
- Для периодов (неделя, месяц) — по-прежнему из `daily_stats`

**3. Бэкфил daily_stats — агрегатные поля из orders_raw:**
- Новая функция `backfill_daily_stats_from_orders_raw()` в `database.py`
- Перебирает все (branch_name, date) из `orders_raw`, вызывает `aggregate_orders_for_daily_stats`, обновляет daily_stats
- Обновляет: `total_delivered`, `late_count`, `late_delivery/pickup_count`, `late_percent`, `avg_late_min`, `avg_cooking/wait/delivery_min`, `cooks/couriers_count`, `discount_types`
- Результат: 3776 пар обновлено за 30 сек

**4. Бэкфил daily_stats — OLAP поля:**
- Новая функция `backfill_daily_stats_olap()` в `database.py`
- Для каждой даты с NULL `cogs_pct` вызывает `get_all_branches_stats` (OLAP v2) и обновляет `cogs_pct`, `sailplay`, `discount_sum`
- Результат: 447 дат обновлено за ~5 мин

**5. Данные по сменам (shifts_raw):**
- `shifts_raw` содержит данные только с 21.02 (Events API — реалтайм, исторических данных нет)
- OLAP v2 не поддерживает `reportType: "EMPLOYEES"` / `"LABOR"` — нет API для исторических смен
- Для дат до 21.02 — `cooks_count`/`couriers_count` = 0, это ожидаемо

### Результат

| Поле | До бэкфила (14.02) | После бэкфила |
|------|-------------------|---------------|
| cogs_pct | NULL | 30.9-32.9% |
| sailplay | NULL | 23-35K ₽ |
| discount_sum | NULL | 52-80K ₽ |
| total_delivered | 500 (=orders_count) | 309 (корректно) |
| avg_cooking_min | NULL | 22-43 мин |
| avg_wait_min | NULL (баг) | 8-13 мин |
| avg_delivery_min | NULL | 25-30 мин |

## 25.02.2026 — Отчёты: формат, /отчёт за период, фильтр точного времени

- Мультисигнальный: комментарий (`точн/тчн`) ИЛИ `planned_time - opened_at > 150 мин`
- Применяется только к расчёту времён (не к опозданиям/выручке)
- Порог 150 мин консервативный: обычный пик-час заказ укладывается в 120-150 мин

**2. Рефакторинг `_format_branch_report`:**
- Убраны зависимости от `rt_data`/`snapshot`
- Данные из `orders_raw` (агрегат) + `shifts_raw` (штат)
- Скидки с количеством: `└ Промокод x 12: 5 000 ₽`
- Блок времён: `⏱ Готовка: 18.5 | Ожидание: 4.2 | В пути: 22.3 мин` + `└ Итого: 45.0 мин`
- Убрана иконка "утро/вечер" — только дата
- Слово "доставок" в строке опозданий

**3. `/день` → `/отчёт` с поддержкой периодов:**
- Команда переименована: `/отчёт [филиал] [период]`
- Парсер периодов: `вчера`, `14.02`, `неделя`, `7д`, `месяц`, `01.02-15.02`, `февраль`

**4. `/статус` — скидки и времена:**
- `aggregate_orders_today` — новая функция, быстрый агрегат из orders_raw за сегодня
- Скидки с количеством из `orders_raw` вместо пустого OLAP `discount_types`
- Блок времён (без итого): готовка, ожидание, в пути

**5. `aggregate_orders_for_daily_stats` — расширен:**
- `total_delivery_count` — для расчёта % опозданий
- `avg_late_min` — среднее время опоздания
- Штат из `shifts_raw` (`cooks_today`, `couriers_today`)
- `COUNT(*)` для типов скидок
- Времена фильтруются через `_EXACT_TIME_FILTER`

## 25.02.2026 — Утренний режим + OLAP enrichment + OpenTime

### Контекст
Вечерние отчёты создавали лишний шум в чате и давали неполные данные (поздние заказы, праздничные дни с +1 час работы). Кроме того, `orders_raw` содержал пустые OLAP-поля для исторических заказов, а `daily_stats` не имел детальных метрик по временам.

### Что сделано

**1. OLAP enrichment pipeline (`olap_enrichment.py` — новый модуль):**
- 1 OLAP v2 запрос per server → 9 полей: `payment_type`, `discount_type`, `source`, `send_time`, `service_print_time`, `cooked_time` (CookingFinishTime), `opened_at` (OpenTime), `pay_breakdown`
- Запись через `UPDATE ... CASE WHEN col IS NULL OR col = '' THEN ? ELSE col END` — не перезатирает существующие данные
- Расписание: 09:00 лок (перед утренним отчётом)
- По понедельникам обогащает за последние 7 дней (покрывает выходные)

**2. OpenTime → opened_at:**
- Обнаружено поле `OpenTime` в OLAP v2 (формат `2026-02-24T11:39:26`)
- Добавлено в ENRICHMENT_FIELDS и _aggregate_by_order
- Бэкфилл за всю историю (с янв 2025): **388 669 заказов**, 374 980 обновлено
- Решило проблему пустого `opened_at` для исторических данных → `avg_cooking_min` теперь работает для всех дат

**3. Расписание перенесено на утро:**

| Было (МСК) | Стало (МСК) | Задача |
|------------|-------------|--------|
| 19:25 | 05:00 | OLAP enrichment |
| 19:30 вс-чт | **УДАЛЕНО** | Вечерний отчёт |
| 20:30 пт/сб | **УДАЛЕНО** | Вечерний отчёт пт/сб |
| 19:50 пт/сб | **УДАЛЕНО** | RT-снапшот |
| 19:31 | 05:26 | iiko → Sheets + ТГ |
| 05:30 | 05:25 | Утренний отчёт |
| 05:31 | 05:27 | Аудит |

Утренний отчёт теперь единственный: OLAP v2 → агрегаты orders_raw → daily_stats → ТГ.

**4. Расширение daily_stats (5 новых столбцов):**
- `late_delivery_count` — опоздания доставки
- `late_pickup_count` — опоздания самовывоза
- `avg_cooking_min` — среднее время готовки (opened_at → cooked_time)
- `avg_wait_min` — среднее ожидание (ready_time → send_time)
- `avg_delivery_min` — среднее время курьера (send_time → actual_time)

**5. Расширение orders_raw (2 новых столбца):**
- `cooked_time` — время завершения готовки (из OLAP CookingFinishTime и Events API)
- `discount_type` — тип скидки per order (из OLAP OrderDiscount.Type)

**6. Фиксы:**
- **Таймзоны:** `opened_at` из Events API хранился с `+07:00`, OLAP-поля без таймзоны. SQLite считал разницу +7 часов. Fix: `replace(substr(opened_at, 1, 19), 'T', ' ')` в SQL.
- **Выбросы avg_cooking_min:** мелкие заказы (30₽ самовывоз — 0.2 мин "готовки"). Fix: `sum >= 200` + `BETWEEN 1 AND 120`.
- **COALESCE-защита upsert_orders_batch:** Events API перезатирал OLAP-поля пустыми значениями. Fix: `COALESCE(NULLIF(excluded.col, ''), orders_raw.col)` для OLAP-полей.

**7. Удалён мёртвый код:**
- Функции `job_send_evening_report`, `job_save_rt_snapshot` из `daily_report.py`
- Импорты `get_branch_rt`, `get_daily_stats`, `get_rt_snapshot`, `save_rt_snapshot`

### Бэкфиллы
- orders_raw enrichment (21-24 фев): 5 423 заказа
- orders_raw OpenTime (вся история): 388 669 заказов
- daily_stats (21-24 фев): 36 записей (9 точек × 4 дня)

### Проверенные OLAP v2 поля (новые)
**Работают:** `OpenTime`, `Delivery.CookingFinishTime`, `Delivery.SendTime`, `Delivery.PrintTime`, `Delivery.SourceKey`, `Delivery.ConfirmTime`, `Delivery.ActualTime`
**Не существуют:** `Delivery.OpenTime`, `OrderTime`, `CreateDate`, `Delivery.CreatedTime`, `Cooking`, `CookingTime`, `LaborCost`, `Employee.Salary`, `Delivery.Duration`, `Delivery.IsLate`

## 24.02.2026 — Баг: отменённые заказы в алертах и поиске

### Проблема
- `/поиск` показывал статус "Новая" для уже отменённых заказов
- Алерты об опоздании продолжали приходить для отменённых заказов

### Расследование (debug mode)
- Выяснено: **iiko Events API никогда не отправляет статус "Отменена"** — это системное ограничение iiko. Статусы в потоке событий: Новая, Не подтверждена, В пути, Доставлена, Закрыта.
- Обнаружен неизвестный тип события `deliveryProblemChanged` — содержит полные данные заказа (статус, адрес, курьер и т.д.), но не обрабатывался.
- Endpoint `/api/v2/deliveries` на BO-серверах iiko Cloud → 404 (недоступен).
- При отмене заказа в iiko: событие не приходит, заказ навсегда остаётся в последнем известном статусе в нашей БД.

### Фикс
1. **`iiko_bo_events.py`**: добавлен обработчик `deliveryProblemChanged` — обновляем статус и поля заказа аналогично `deliveryOrderEdited`.
2. **`late_alerts.py`**: заказы с `overdue_min > 30` + без курьера + статус "Новая"/"Не подтверждена" → пропускаем. Логика: реальные опоздавшие заказы всегда имеют курьера к этому моменту; без курьера = отменён или брошен.
3. **`arkentiy.py`** (`/поиск`): заказы с активным статусом старше 1 часа получают пометку "⚠️ статус может быть устарел — iiko не прислала обновление".

### Решение: OLAP v2 polling (cancel_sync)

**Прорыв:** обнаружен рабочий endpoint `POST /api/v2/reports/olap?key=TOKEN` — принимает JSON body, возвращает JSON (не XML). Поле `Delivery.CancelCause` содержит причину отмены для отменённых заказов (`null` для остальных).

**Новый модуль `cancel_sync.py`:**
- Запускается каждые 3 минуты (APScheduler IntervalTrigger)
- Запрашивает OLAP v2 по всем 9 серверам (группируя по bo_url — 5 уникальных серверов → 5 запросов)
- Фильтрует заказы где `Delivery.CancelCause != null`
- Обновляет `orders_raw.status = 'Отменена'`, `cancel_reason = причина` (UPDATE WHERE status != 'Отменена')
- Обновляет in-memory `_states` — ставит статус 'Отменена' (не удаляет из dict — чтобы `/поиск` работал)
- Первый запуск: 22 заказа обновлено

**Убрана debug-инструментация из `iiko_bo_events.py`** (гипотезы B и AC — подтверждены, логи больше не нужны).

**Эффект:**
- `/поиск` показывает реальный статус «Отменена» + причину
- `late_alerts` не отправляет алерты для отменённых (status != 'Новая'/'В пути')
- Задержка определения — до 3 мин (частота polling). Events API по-прежнему не присылает статус "Отменена"
- Причина отмены (`cancel_reason`) сохраняется в `orders_raw` для аналитики
