## Сообщение

Состоит из двух блоков: **результат загруженного реестра** + **проблемы, требующие внимания**.

### Принципы

1. **Показывать только проблемы.** Если всё ок — одна строка. Не грузить "✅ 13 зак." по каждому дню.
2. **Группировка по срочности.** Сначала просрочено (>4 дн), потом свежее.
3. **Каждый эмодзи = одно значение.** Не смешивать статус дня с типом проблемы.
4. **Меньше цифр, больше смысла.** "5 заказов на 8 200 р" вместо "5 ❓ (8 200 р) — 3 дн."

### Основное состояние (всё чисто)

```
💳 Реестр ТБанк · 22.02.2026
98 заказов · 190 725 р · комиссия 3 067 р
✅ Все подтверждены

📋 Онлайн-оплаты · с 19.02
✅ Все оплаты подтверждены
```

### Основное состояние (есть проблемы)


## Логика отображения блока "Онлайн-оплаты"

### Что показывать:

1. **Итоговая строка** — всего ожидающих заказов + сумма. Если 0 → "✅ Все оплаты подтверждены"
2. **Блок 🔴 Просрочено** — pending > 4 дней. Группировка: точка · дата. С перечислением конкретных заказов (до 3 на группу).
3. **Блок ⏳ Ожидают реестр** — pending ≤ 4 дней. Только счётчики, без конкретных заказов (их ещё рано тревожить).

### Что НЕ показывать:

- Дни где всё ✅ — не нужно "21.02 ✅ 13 зак." Если проблем нет = молчим.
- Число confirmed — не информативно ("120 ✅" — ну и что?).
- "нет в iiko" как отдельную категорию в статусе — это деталь реестра, не статуса.
- Сегодняшнюю дату в "ожидают" — реестра за сегодня по определению нет.

## Кнопки

### Состояние 0 — Основной отчёт

Кнопка показывается **только если есть pending-заказы** (overdue или ожидают реестр):

```
[🔍 Детализация по точке]
```

| Кнопка | callback_data | Действие |
|--------|--------------|----------|
| 🔍 Детализация по точке | `tbank:branches` | Редактировать то же сообщение → Состояние 1 |

---

### Состояние 1 — Выбор точки

Сообщение полностью заменяется на:

```
📋 Онлайн-оплаты · детализация

Выбери точку:

[Барнаул_1 Ана  ⏳ 16 зак.]
[Барнаул_2 Гео  ⏳ 14 зак.]
[Барнаул_3 Тим  🔴 1 · ⏳ 11 зак.]
[Барнаул_4 Бал  ⏳ 12 зак.]

[← Назад к отчёту]
```

- Кнопка на каждую точку с pending — одна кнопка в ряд
- Иконка: 🔴 если есть overdue, ⏳ если только ожидают; счётчик заказов
- Сортировка: сначала точки с 🔴, потом ⏳, внутри — по убыванию заказов

| Кнопка | callback_data | Действие |
|--------|--------------|----------|
| `[Барнаул_3 Тим  🔴 1 · ⏳ 11 зак.]` | `tbank:branch:{branch_id}` | Редактировать → Состояние 2 |
| ← Назад к отчёту | `tbank:back` | Редактировать → Состояние 0 (восстановить исходный отчёт) |

---

### Состояние 2 — Детализация по точке

Сообщение заменяется на детальный список:

```
📋 Барнаул_3 Тим · ожидают подтверждения

🔴 Просрочено:
20.02 — 1 заказ · 1 546 р
  └ #134010: 1 546 р

⏳ Ожидают реестр:
24.02 — 11 заказов · 19 707 р
  └ #134001: 1 800 р
  └ #134002: 2 300 р
  └ #134003: 1 650 р
  └ #134004: 1 900 р
  └ ... ещё 7

[← К точкам]   [← К отчёту]
```

- Блоки: сначала 🔴 Просрочено, потом ⏳ Ожидают реестр
- Внутри каждой даты: показать до 5 заказов, если больше — строка `└ ... ещё N`
- Заказы без суммы-уточнений — просто `#номер: сумма р`

| Кнопка | callback_data | Действие |
|--------|--------------|----------|
| ← К точкам | `tbank:branches` | Редактировать → Состояние 1 |
| ← К отчёту | `tbank:back` | Редактировать → Состояние 0 |

---

### Пустое состояние детализации

Если у точки pending = 0 (вдруг сбросилось пока смотрел):

```
📋 Барнаул_1 Ана

✅ Нет ожидающих заказов

[← К точкам]
```

## Технические ограничения

- Длина сообщения: < 4096 символов. При > 4096 — урезать до топ-5 дат в блоке ⏳.
- `parse_mode`: `HTML`
- **Редактирование:** все переходы — `editMessageText` + `editMessageReplyMarkup`, новых сообщений не создавать
- `callback_data` max 64 байта — использовать короткие branch_id из конфига
- Изображения/файлы: нет

## Зависимости

- Данные из: `online_payments` таблица, iiko OLAP v2
- Существующие функции: `get_tracking_summary`, `get_pending_payments`, `get_overdue_payments`
- Нужен callback handler: `tbank:back`, `tbank:branches`, `tbank:branch:{branch_id}`
- Исходный текст отчёта (Состояние 0) должен сохраняться для возврата — либо регенерируется из БД, либо кешируется

## Заметки для @интегратора

- Сегодняшние pending (order_date = today) — не показывать в блоке "⏳ Ожидают реестр", это шум
- Блок "🔴 Просрочено" — сортировка по дате (старые сверху)
- Блок "⏳ Ожидают" — сортировка по количеству заказов (больше = выше)
- Если pending всего 0 и просрочено 0 — весь блок "Онлайн-оплаты" = одна строка "✅", кнопка Детализация не показывается
- `branch_id` в callback_data = ключ из `tbank_branches.json` (гарантированно < 64 байт)
- Состояние 0 при возврате — проще всего регенерировать из БД на лету (актуальнее, чем кеш)
