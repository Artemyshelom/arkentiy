# Аудит проекта Аркентий

Дата: 2026-03-10

Формат: read-only аудит без правок кода.

## Что проверено

- Правила и документация: `rules/integrator/architecture.md`, `rules/integrator/database.md`, `rules/integrator/lessons.md`, `README.md`, `PROJECT_UPDATE_STATUS.md`, профильные docs/specs.
- Критичный backend: `app/main.py`, `app/config.py`, `app/ctx.py`, `app/db.py`, `app/database_pg.py`.
- Мультитенантность и realtime: `app/clients/iiko_bo_events.py`, `app/routers/stats.py`.
- Auth / cabinet / onboarding / payments: `app/routers/auth.py`, `app/routers/cabinet.py`, `app/routers/onboarding.py`, `app/routers/payments.py`, `app/services/auth.py`, `app/clients/email.py`.
- OLAP и интеграции: `app/clients/iiko_auth.py`, `app/clients/iiko_bo_olap_v2.py`, `app/clients/olap_queries.py`.
- Web second pass: `web/reset-password.html`, `web/verify.html`, `web/register.html`, `web/js/wizard.js`, `web/js/cabinet.js`, `web/payment/success.html`, `web/payment/fail.html`, `web/cabinet/subscription.html`, `web/cabinet/billing.html`, `web/cabinet/connections.html`.

## Короткий вывод

Система рабочая по общему направлению, но до уровня аккуратного SaaS на 50+ tenants не дотягивает. Основная проблема не в синтаксисе и не в том, что проект не запускается, а в смешении целевой многотенантной архитектуры с унаследованными single-tenant fallback-паттернами.

По итогам аудита я не согласен с частью документации и статусных записей в той форме, будто многотенантность уже аккуратно доведена до конца. В коде остаются дефолты на `tenant_id = 1`, глобальные realtime-структуры без tenant-scoping, реальные contract-баги между frontend и backend, а также утечки секретов в документации и рабочем дереве.

Итог: правки нужны. Часть из них обязательна до дальнейшего масштабирования, часть желательна до активного роста веб-кабинета.

## Критичные риски

### 1. В проекте есть реальные секреты, и часть из них попала в документацию

Почему это критично:

- Для SaaS это прямой security-риск и риск утечки данных.
- Проблема уже не теоретическая: в репозитории присутствуют реальные значения токенов в рабочем дереве и в документации.

Подтверждения:

- `secrets/api_keys.json` содержит действующий-looking key для `boris_agent` и `tenant_id: 1`.
- `.gitignore` исключает `secrets/`, но сам файл физически существует в workspace, значит секрет уже распространяется через рабочие копии.
- `docs/specs/boris_api_prompt.md` содержит Bearer token целиком.
- `docs/journal.md` содержит тот же токен и примеры запросов с ним.

Риск:

- Любой, кто получил рабочую копию проекта или экспорт документации, получает рабочий доступ к Stats API.
- Это бьёт и по безопасности, и по соответствию заявленному правилу «не должно быть секретов в коде». 

Нужно править:

- Обязательно.
- Ротация токенов.
- Вычистить секреты из docs и рабочих файлов.
- Перевести примеры в docs на placeholders.

### 2. Мультитенантность не доведена до безопасного уровня

Почему это критично:

- Сейчас проект рассчитан на 2 tenants, но пользовательский сценарий уже требует 50+.
- При таком масштабе silent fallback на tenant 1 будет приводить либо к утечкам, либо к неверной аналитике.

Подтверждения:

- `app/ctx.py`: `ctx_tenant_id` имеет `default=1`.
- `app/config.py`: свойство `branches` в PG-режиме читает кэш через `_get_branches(1)`.
- `app/routers/stats.py`: `tenant_id: int = token_meta.get("tenant_id", 1)`.
- `app/database_pg.py`: большое количество функций объявлено как `tenant_id: int = 1`.
- `app/database_pg.py`: `aggregate_orders_for_daily_stats(branch_name, date_iso)` вообще не принимает `tenant_id` и фильтрует только по `branch_name` и дате.
- `app/database_pg.py`: `get_payment_changed_orders(branch_names, date_iso)` тоже без `tenant_id`.
- `app/clients/iiko_bo_events.py`: глобальный realtime state хранится как `_states: dict[str, BranchState] = {}`, то есть индексируется только по `branch_name`.
- `app/clients/iiko_bo_events.py`: fallback-путь вызывает `close_stale_shifts(..., tenant_id=1 по умолчанию)` и `_seed_sessions_from_db(state)` без явного tenant scope.

Риск:

- Один и тот же `branch_name` у разных клиентов теоретически пересечёт realtime.
- Часть fallback-логики будет работать по owner-tenant даже если запрос пришёл для другого клиента.
- Суточные и служебные агрегаты могут смешивать данные одинаковых branch names между tenants.

Нужно править:

- Обязательно до роста на десятки tenants.
- Везде убрать implicit tenant defaults для бизнес-функций.
- Realtime state индексировать по `(tenant_id, branch_name)`.
- Для критичных агрегаций и fallback-функций tenant должен быть обязательным аргументом.

### 3. Сброс пароля сейчас сломан на стыке email -> web -> backend

Почему это критично:

- Это пользовательский auth flow. Если он сломан, часть клиентов просто не восстановит доступ.

Подтверждения:

- `app/clients/email.py` отправляет ссылку вида `/reset-password?token=...`.
- Реальная статическая страница называется `web/reset-password.html`.
- `app/routers/auth.py`: `ResetPasswordRequest` ожидает поля `token` и `password`.
- `web/reset-password.html` отправляет `{ token, new_password: password }`.

Риск:

- Письмо ведёт не на тот URL.
- Даже если страницу открыть вручную, backend ждёт другое имя поля.
- В результате reset flow ломается до входа в кабинет.

Нужно править:

- Обязательно.

### 4. Повторная оплата после неуспешного платежа сломана

Почему это критично:

- Это прямой путь потери выручки.

Подтверждения:

- `web/payment/fail.html` делает `POST /api/payments/create` без JWT Authorization header.
- `app/routers/payments.py` в `api_create_payment()` получает `tenant_id` только через `Depends(get_tenant_id)` из JWT.
- Frontend при этом передаёт `tenant_id` в body, но backend его не использует для авторизации.

Риск:

- Retry после неуспешной оплаты не сработает для пользователя без активного JWT в браузере.
- Пользователь попадает в тупик «зарегистрируйтесь заново».

Нужно править:

- Обязательно.

## Существенные проблемы

### 5. Cabinet `test_iiko` реализован с реальным логическим дефектом

Подтверждения:

- `app/routers/cabinet.py` выбирает из БД только `bo_url, bo_login`.
- Далее пытается читать `cred.get("bo_password")`.
- Возвращает жёстко прошитый `response_time_ms: 500`, а не фактическое время.

Следствие:

- Проверка подключения либо работает некорректно, либо даёт ложное представление о состоянии интеграции.

Нужно править:

- Да, в ближайшем цикле.

### 6. Настройки Telegram-чата в cabinet теряют многогородной смысл

Подтверждения:

- `web/cabinet/connections.html` оперирует массивом `cities`.
- `app/routers/cabinet.py` в `update_chat()` сохраняет только `city = data.cities[0] if data.cities else None`.

Следствие:

- UI создаёт впечатление, что чат можно связать с набором городов.
- Backend хранит только один город.

Нужно править:

- Да. Это не security, но это реальный функциональный contract drift.

### 7. Onboarding iiko test слишком упрощён и может давать ложные результаты

Подтверждения:

- `app/routers/onboarding.py` тестирует iiko через `login + SHA1("")`.
- Визард вообще не собирает BO password, только URL и login.
- При создании tenant `app/routers/onboarding.py` пишет в `iiko_credentials` только `bo_url` и `bo_login`, без `bo_password`.

Следствие:

- Тест подключения может быть «успешен» или «неуспешен» по логике, не совпадающей с реальным рабочим режимом интеграции.
- Для инсталляций, где нужен нормальный пароль, onboarding не подготавливает данные для production-подключения.

Нужно править:

- Да, до масштабирования онбординга на внешних клиентов.

### 8. В проекте есть дублирование OLAP-слоя и размытая точка истины

Подтверждения:

- `app/clients/iiko_bo_olap_v2.py` и `app/clients/olap_queries.py` решают близкие задачи.
- Оба слоя работают с BO auth, группировкой точек и HTTP-запросами в OLAP.
- Документация местами описывает новую каноничную схему, но runtime продолжает использовать оба варианта.

Следствие:

- Повышенная стоимость поддержки.
- Риск, что баги и правки будут расходиться между двумя слоями.

Нужно править:

- Да, но после закрытия критичных рисков.

### 9. SSL verification отключена во многих внешних HTTP-клиентах

Подтверждения:

- `app/routers/onboarding.py`: `httpx.AsyncClient(verify=False, timeout=15)`.
- `app/clients/iiko_auth.py`, `app/clients/iiko_bo_olap_v2.py`, `app/clients/olap_queries.py`, `app/clients/iiko_bo_events.py`, часть onboarding backfill-скриптов делают то же самое.

Следствие:

- Для внутренних контуров это иногда осознанный компромисс, но для SaaS это всё равно понижает базовую сетевую безопасность.

Нужно править:

- Да. Хотя это не обязательно блокер dev-среды, для production-политики это плохая норма.

## Неэффективности и архитектурные долги

### 10. В коде много owner-centric fallback-паттернов

Примеры:

- `tenant_id = 1` как дефолт в DB API.
- Конфиг точек через `settings.branches`, который в PG-режиме всё равно смотрит на tenant 1.
- Часть realtime fallback-логики тянет данные не из текущего tenant context, а из общего/дефолтного слоя.

Следствие:

- Даже если на тестовых данных всё «как будто работает», архитектура уже нестабильна для 50+ tenants.

### 11. Есть признаки лишнего расхода запросов и сопровождения из-за раздвоенной логики

Примеры:

- Realtime + DB fallback + OLAP + отдельные старые job-пайплайны местами перекрывают друг друга.
- Два OLAP-клиента дублируют сетевую и бизнес-логику.
- Часть кабинета рисует интерфейс шире фактической модели backend, из-за чего растёт стоимость поддержки фронта и API.

### 12. Документация завышает уровень завершённости отдельных зон

Наблюдение:

- В правилах и docs направление выбрано в целом верное.
- Но по факту кодовая база ещё не соответствует заявленной аккуратной многотенантной модели и безопасной дисциплине секретов.

Итог:

- Документация полезна как target architecture.
- Как описание текущего состояния системы она местами слишком оптимистична.

## Второй проход: web / cabinet / payments / onboarding API

Ниже только подтверждённые наблюдения именно по второму проходу UI ↔ API.

### A. Что совпадает нормально

- `web/js/wizard.js` после создания tenant сохраняет JWT в `localStorage`, и дальше запросы на `/api/invoices/create` и `/api/payments/create` уже идут с Bearer token. Этот конкретный happy-path в визарде выглядит логично.
- `web/cabinet/subscription.html` вызывает `/api/cabinet/subscription/history`, `/subscription/change-plan`, `/subscription/change-payment-method`; эти endpoints в `app/routers/cabinet.py` действительно существуют.
- `web/cabinet/billing.html` и `web/cabinet/connections.html` используют существующие cabinet endpoints.

### B. Подтверждённые contract-баги

#### B1. Reset password: frontend и backend говорят на разных payload schema

- Backend: `app/routers/auth.py` ждёт `password`.
- Frontend: `web/reset-password.html` шлёт `new_password`.

Итог:

- Broken API contract.

#### B2. Reset password: email ведёт не на существующий публичный маршрут страницы

- Email client отправляет `/reset-password?token=...`.
- Статический файл лежит как `reset-password.html`.

Итог:

- Broken entrypoint для пользователя.

#### B3. Payment retry page не соблюдает auth contract backend

- `web/payment/fail.html` делает anonymous POST на `/api/payments/create`.
- Backend требует JWT via `Depends(get_tenant_id)`.

Итог:

- Broken retry flow.

#### B4. iiko test в cabinet потенциально ложно проверяет существующее подключение

- Frontend корректно зовёт `/api/cabinet/connections/iiko/test`.
- Но backend берёт не все нужные поля и отдаёт искусственный `response_time_ms`.

Итог:

- UI может показывать ложную исправность интеграции.

#### B5. Chat settings UI шире фактической модели хранения

- Frontend работает с массивом городов.
- Backend сохраняет только первый город.

Итог:

- Contract drift между UX и persistence-моделью.

### C. Риски проектных решений на web-слое

#### C1. JWT хранится в `localStorage`

Подтверждения:

- `web/js/cabinet.js` читает и пишет `arkentiy_token` в `localStorage`.
- `web/js/wizard.js`, `web/login.html`, `web/reset-password.html` используют ту же схему.

Комментарий:

- Для быстрого SPA/static-кабинета это распространённый компромисс, но с точки зрения security это слабее, чем HttpOnly cookie.
- Это не самый срочный дефект на фоне broken flows и tenant leakage, но долг зафиксировать стоит.

#### C2. Часть бизнес-ценообразования дублируется на клиенте

Подтверждения:

- `web/js/wizard.js` локально пересчитывает pricing, скидки, annual discount, promo бонусы.
- Backend при создании tenant пересчитывает цену отдельно.

Комментарий:

- Это не баг само по себе, если backend остаётся единственным источником истины.
- Но при развитии тарифов велик риск расхождения UI preview и фактического счёта.

## Где я согласен с текущей архитектурой

- Вектор на `tenant_id` + отдельные tenant tables / modules / chats / subscriptions выбран правильно.
- Переход на PostgreSQL вместо локального single-DB режима правильный.
- Использование отдельного cabinet API и отделённого статического web-слоя логично для текущей стадии продукта.
- Визард onboarding в целом неплохо продуман как пользовательский сценарий.

## Где я не согласен с текущими заявлениями о готовности системы

- Нельзя считать многотенантность завершённой, пока в коде остались дефолты на tenant 1 и глобальные realtime maps без tenant scoping.
- Нельзя считать security-практики нормальными, пока в docs и рабочем дереве лежат реальные токены.
- Нельзя считать web-кабинет доведённым, пока broken reset flow и broken payment retry уже подтверждаются чтением кода без предположений.

## Приоритет исправлений

### Обязательно

- Удалить секреты из docs и рабочих файлов, перевыпустить токены.
- Убрать implicit fallback на `tenant_id = 1` из критичных бизнес-путей.
- Перевести realtime state на tenant-scoped ключи.
- Исправить reset-password flow целиком.
- Исправить payment retry flow.
- Исправить cabinet iiko test.

### Высокий приоритет

- Привести onboarding iiko model к реальному production-сценарию.
- Убрать contract drift по Telegram chat cities.
- Пересмотреть функции без tenant filter в `app/database_pg.py`.

### Средний приоритет

- Свести OLAP к одному каноничному слою.
- Снизить число `verify=False` и оформить осознанную политику TLS.
- Синхронизировать docs с реальным текущим состоянием.

## Итоговое решение по необходимости правок

Править нужно.

- Без правок проект остаётся рабочим для ограниченного круга текущих tenants, но уязвимым и хрупким для SaaS-масштабирования.
- Критичные правки не косметические: это не «полирнуть код», а закрыть реальные security, multitenancy и revenue-impact проблемы.
- После исправления критики архитектура может стать хорошей базой для роста. Сейчас она скорее в состоянии «сильный прототип / ранний production», чем «аккуратно подготовленный SaaS foundation».
