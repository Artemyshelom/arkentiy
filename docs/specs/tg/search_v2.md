| Проблема | Что сейчас | Что должно быть |

| Пачка сообщений | При N филиалах — N+1 отдельных сообщений | Одно сообщение на весь поиск |

## Карточка заказа — полная (детальный формат)

Используется в сценариях A, B, C, D при открытии конкретного заказа.

**Формат строки суммы + оплаты:**
```
💰 2 691 ₽ · Наличные
💰 1 423 ₽ · Онлайн
💰 3 400 ₽ · Безналичный расчёт
💰 —       · (если сумма неизвестна)
```

**Маппинг payment_type → читаемое название:**

| Значение в БД | Отображение |
|--------------|-------------|
| `Наличные` | Наличные |
| `Безналичный расчет` / `Безналичный расчёт` | Безнал |
| `Онлайн` / `Тинькофф` / `Т-Банк` | Онлайн |
| `Системы лояльности` | Бонусы |
| пусто / null | *(не показывать)* |

**Формат строки опоздания в компактном списке:**
- `✅` — вовремя (без доп. текста)
- `🔴 +37 мин` — опоздал, показываем минуты
- `⏳` — ещё не доставлен
- `—` — отменён

## Кнопки — полная спецификация

| Кнопка | callback_data | Действие |
|--------|--------------|---------|
| `📋 Барнаул_4 Бал` | `srch:{num}:{msg_id}:{branch_idx}` | edit_message: карточка этого филиала |
| `#119636` | `srch:{num}:{msg_id}` | edit_message: карточка заказа |
| `← #85191 (все филиалы)` | `srch:back:{msg_id}` | edit_message: сводный список |
| `← К клиенту` | `srch:back:{msg_id}` | edit_message: список заказов клиента |
| `← К результатам` | `srch:back:{msg_id}` | edit_message: список результатов |

> **msg_id** = Telegram message_id сообщения с результатами поиска — передаётся в callback чтобы знать что редактировать и куда возвращаться.

## Технические ограничения

- Длина сообщения: < 4096 символов
- `parse_mode`: `HTML`
- **Редактирование сообщений (edit_message):** да — основной механизм навигации
- Изображения/файлы: нет
- Лимит callback_data: 64 байта. `srch:119636:1234567890` ≈ 25 байт — в пределах нормы.
- Inline keyboard: кнопки по 3 в ряд для списков, по 2-4 для мульти-филиала

## In-memory кэш для навигации назад

Для кнопки "← Назад" нужно хранить предыдущее состояние (текст + клавиатура).

**Структура кэша:**
```python
_search_cache: dict[tuple[int, int], dict] = {}
# (chat_id, message_id) -> {"text": str, "keyboard": list}
```

**TTL:** очищать записи старше 30 минут (или при перезапуске бота — ок).

**Размер:** кэш небольшой, в памяти храним только последние N поисков (ограничить 100 записями).

## Заметки для @интегратора

### Что меняется в коде:

**1. `_handle_search` — возвращает message_id отправленного сообщения**
- При единственном результате: отправить карточку без header и без кнопок
- При нескольких результатах по номеру: одно сводное сообщение + кнопки
- При результатах по телефону: имя клиента в шапке, список без дублирования, кнопки

**2. Новый in-memory кэш `_search_cache`**
```python
_search_cache: dict[tuple[int, int], dict] = {}  # (chat_id, msg_id) -> {text, keyboard}
```
Сохранять при отправке, читать при `srch:back:{msg_id}`.

**3. Callback `srch:` — новые обработчики (вместо/параллельно `order:`)**
- `srch:{num}:{msg_id}` — открыть карточку конкретного заказа, edit_message
- `srch:{num}:{msg_id}:{branch}` — открыть карточку конкретного филиала, edit_message
- `srch:back:{msg_id}` — вернуться, edit_message из кэша

> **Старый callback `order:` можно оставить** для обратной совместимости (там где уже есть кнопки), но новые поиски генерировать с `srch:`.

**4. `_format_order_card` — добавить payment_type**
- В строку суммы: `💰 2 691 ₽ · Наличные` (если payment_type есть)
- Добавить `payment_type` в COLS запроса

**5. `_format_order_compact` — добавить время опоздания**
- Вместо просто `🔴` → `🔴 +{late_minutes:.0f} мин`
- Добавить `late_minutes` в COLS запроса (уже есть)

**6. Логика определения "поиск по телефону"**
- Если запрос начинается с `+` или `7` и 10-11 цифр — режим телефона
- Режим телефона: имя клиента в шапке, кнопки с номерами заказов

**7. Граничный случай: один результат при поиске по телефону**
- Клиент сделал 1 заказ → сразу карточка (без списка)

**8. Лимит строк кэша**
- Если `len(_search_cache) > 100` → удалить самую старую запись (FIFO)
