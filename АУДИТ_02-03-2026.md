## ✅ Что сделано хорошо — сохранить и не трогать

Прежде чем перейти к проблемам — проект на высоком инженерном уровне. Это production SaaS с 9 точками, real-time обработкой и мультитенант-архитектурой.

1. **[А1] Модульная структура** — чёткое разделение на `clients/`, `jobs/`, `routers/`, `services/`, `monitoring/`, `webhooks/`. Каждый модуль = отдельный файл. ✓
2. **[А2] main.py** — только импорты, `register_jobs()` и lifespan. Никакой бизнес-логики. Каждый job отключается одной строкой. ✓
3. **[А5] Модули отключаются одной строкой** — закомментированные `scheduler.add_job(...)` для стоп-листов, Битрикс24, MyMeet. ✓
4. **[А7] Конфигурация из branches.json** — нет хардкода городов/точек в коде. Всё через `settings.branches`. ✓
5. **[Б2] Секреты через os.getenv** — все токены, API-ключи, пароли из `.env`. Pydantic Settings корректно. ✓
6. **[Б7] .gitignore** — `.env`, `secrets/`, `data/`, `*.db` — всё защищено. ✓
7. **[Б8] Параметризованные SQL-запросы** — asyncpg `$1, $2`, aiosqlite `?`. Конкатенации строк в SQL нет. ✓
8. **[Ч1] Events API сортируются** — из `Уроки_и_баги.md` это уже известный паттерн, реализован. ✓
9. **[Ч2] deliveryOrderEdited = merge** — обновляются только пришедшие поля, не overwrite. ✓
10. **[Э1] Токены iiko кэшируются** — `iiko_tokens` таблица с TTL, `iiko_auth.py` с кэшем 3000 сек. ✓
11. **[Э5] Telegram с retry** — exponential backoff, rate limit handling (429), split длинных сообщений. ✓
12. **[Э6] UPSERT** — INSERT OR REPLACE / ON CONFLICT используется повсюду. ✓
13. **[П2] Нет time.sleep()** — используется только `asyncio.sleep()`. ✓
14. **[П1] APScheduler async** — AsyncIOScheduler, все jobs async. ✓
15. **[П5] Logging на уровне INFO** — `settings.log_level` по умолчанию `INFO`, не DEBUG. ✓
16. **Real-time архитектура BranchState** — event sourcing, in-memory состояние, fallback из shifts_raw при рестарте. Инженерно грамотно.
17. **Мультитенант ContextVar** — `ctx_tenant_id`, per-tenant polling loops. Продуманная архитектура масштабирования.
18. **Healthcheck + backup** — Docker HEALTHCHECK + автобэкап SQLite на Google Drive ежедневно.

### 🔴 [Б3] Токены iiko передаются в URL query string

```python
# Вместо:
from app.database_pg import _pool
pool = _pool

# Использовать:
from app.db import get_pool
pool = get_pool()
```

### 🟡 [Э3] Нет retry-логики на OLAP v2 запросы

**Файл:** `app/clients/iiko_bo_olap_v2.py:69-78`
```python
resp = await client.post(
    f"{bo_url}/api/v2/reports/olap?key={token}",
    json=body,
    headers={"Content-Type": "application/json"},
    timeout=30,
)
if resp.status_code != 200:
    logger.warning(...)
    return []
```

**Проблема:** При 500/502/503 от iiko BO сервера запрос просто логируется и возвращает пустой список. Утренний отчёт выйдет с нулями вместо данных. iiko BO нестабилен (из `Уроки_и_баги.md`).

**Исправление:** Добавить retry с backoff (1-2 повтора):
```python
for attempt in range(3):
    resp = await client.post(...)
    if resp.status_code == 200:
        return resp.json().get("data", [])
    if resp.status_code in (500, 502, 503):
        await asyncio.sleep(2 ** attempt)
        continue
    break
return []
```

### 🟡 [П3] OLAP v2 клиент без timeout на уровне AsyncClient

**Файл:** `app/clients/iiko_bo_olap_v2.py:105, 218, 271`
```python
async with httpx.AsyncClient(verify=False) as client:
```

**Проблема:** `httpx.AsyncClient()` без `timeout=` использует дефолт httpx (5 сек на connect, без общего таймаута). Внутри `_query_olap_v2` передаётся `timeout=30`, но другие запросы к этому клиенту (если добавятся) не защищены.

**Исправление:** Установить timeout при создании клиента:
```python
async with httpx.AsyncClient(verify=False, timeout=30) as client:
```

### 🟡 [Э5] Telegram-сообщения через прямой HTTP, не через очередь

**Файл:** `app/clients/telegram.py`

**Проблема:** Каждое сообщение создаёт новый `httpx.AsyncClient` (строка 73). При большой нагрузке (много алертов одновременно):
- Множество параллельных TCP-соединений к api.telegram.org
- Легко получить 429 rate limit
- Нет приоритизации (алерт опоздания vs мониторинг)

**Исправление (при масштабировании):**
- Использовать `asyncio.Queue` для централизованной отправки
- Один persistent `httpx.AsyncClient` с connection pooling
- Приоритетная очередь: 🔴 алерты > 📊 отчёты > 🔔 мониторинг

1. `docker compose build --no-cache && docker compose up -d`
2. `docker compose logs -f --tail=100 app` — проверить что нет ошибок при старте
3. Проверить `/health` — 200 OK
4. Проверить Telegram: "🟢 Аркентий запущен" пришло в мониторинг
5. Проверить что `/run/late_alerts` возвращает 403 (аутентификация работает)
6. Дождаться одного цикла Events API (30 сек) — проверить что данные поступают
7. Отправить `/статус` в Telegram — проверить что бот отвечает
